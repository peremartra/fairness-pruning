# Bias Benchmarks Zeroed

This directory stores post-intervention evaluation artifacts for fairness-pruned (zeroed) model variants.

The files here are generated by the zeroed-evaluation notebooks documented in notebooks/README.md:

- 07_EvalPrunedModels.ipynb (English BBQ and capability checks on zeroed Llama-3.2-1B)
- 08_EvalLlamaPrunedEsp.ipynb (Spanish EsBBQ on zeroed Llama-3.2-1B)
- 08_EvalSalamandraPrunedEsp.ipynb (Spanish EsBBQ on zeroed Salamandra-2B)

This README describes file contents and generation pipeline only. It does not include interpretation of metric values.

## Common schemas

### 1) Zeroed results CSV files

All three CSV files use the same column layout:

- model
- model_key
- experiment_id
- experiment_name
- n_neurons_zeroed
- eval_limit
- task
- acc
- acc_stderr
- accuracy_amb
- accuracy_disamb
- amb_bias_score
- disamb_bias_score

Notes:

- task identifies the evaluated benchmark task (bbq or one EsBBQ category task such as esbbq_age).
- For EsBBQ CSV exports, some ambiguous/disambiguated fields may appear as N/A in the flattened export depending on the task-level metric mapping used during consolidation.

### 2) Zeroed results JSON files

All zeroed_results.json files are hierarchical and start with:

- metadata
- experiments

metadata includes runtime and run descriptors such as:

- model_name
- model_key
- eval_type
- eval_limit
- started_at
- completed
- language (present in EsBBQ files)

experiments is a dictionary keyed by experiment id (for example exp1, exp2, ...).
Each experiment contains:

- experiment_name
- n_neurons
- results

results is task-keyed:

- BBQ file: task key bbq
- EsBBQ files: one key per category task (esbbq_age, esbbq_gender, esbbq_religion, etc.)

Each task object stores metric fields emitted by lm-evaluation-harness and post-processing in the notebook.

- BBQ includes global and category-level bias fields (for example amb_bias_score_* and disamb_bias_score_*).
- EsBBQ includes acc, acc_norm, acc_ambig, acc_disambig, bias_score_ambig, bias_score_disambig and companion stderr fields.

### 3) Neuron manifest JSON

llama-3.2-1B_neuron_manifest.json stores the reproducibility manifest for the Llama zeroing experiments.

Top-level keys:

- model
- model_key
- language
- generated_at
- experiments

Each experiments entry (exp1 to exp6) includes:

- name
- categories
- score_type
- threshold_pct
- total_neurons
- by_layer

by_layer maps each layer index to the list of neuron indices zeroed in that experiment.

### 4) Generation sanity-check JSON files

Each llama-3.2-1B_expN_generation.json file stores qualitative before/after text generations.

Top-level keys:

- model
- experiment
- exp_id
- n_neurons
- prompts

prompts is a list of prompt records with:

- prompt
- base_model
- zeroed_model

There are 7 prompts per experiment file in this directory.

## How these files are generated

1. Experiment design and neuron selection
- The notebook computes neuron_indices per experiment from precomputed neuron analysis artifacts under results/neuron_analysis.
- It writes an experiment manifest (llama-3.2-1B_neuron_manifest.json) before running evaluations.

2. Model intervention
- For each experiment, the notebook applies zero_neurons_mlp to the selected MLP neurons.

3. Sanity-check generations
- For Llama experiments in notebook 07, base vs zeroed generations are produced and exported as llama-3.2-1B_expN_generation.json.

4. Benchmark evaluation
- Notebook 07 evaluates zeroed Llama variants on BBQ (English).
- Notebook 08 (Llama) evaluates zeroed Llama variants on EsBBQ (Spanish).
- Notebook 08 (Salamandra) evaluates zeroed Salamandra variants on EsBBQ (Spanish).

5. Consolidation
- Each notebook exports a full JSON structure and a flattened CSV table for downstream analysis.

## Usage notes

- Use CSV files for quick plotting and table joins.
- Use JSON files when you need the complete experiment/task hierarchy and full metric fields.
- Use the neuron manifest to reproduce exactly which neurons were zeroed in each experiment.
